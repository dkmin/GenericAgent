GenericAgent는 외부 기여를 환영합니다. 다음 흐름을 따르면 PR이 빠르게 머지됩니다.

## 1. 이슈부터 — 또는 디스커션

```text
새 기능 / 큰 리팩토링  →  먼저 GitHub Issue 또는 Discussion 으로 의도 확인
작은 버그 / 오타       →  바로 PR 가능
```

## 2. 브랜치

```bash
git clone https://github.com/dkmin/GenericAgent.git
cd GenericAgent
git checkout -b feat/<짧은-키워드>
# 또는: fix/<버그>, docs/<페이지>, refactor/<영역>
```

## 3. 로컬 검증

```bash
cp mykey_template_en.py mykey.py
# (mykey.py 에 본인 API Key 채움)
pip install -e ".[ui]"
python launch.pyw
```

## 4. 코드 스타일

- 기존 파일 스타일을 따르세요. 한 파일 안에서는 일관되게.
- **Atomic Tools 추가 금지** — 9개로 충분합니다. 새 능력은 SOP/스킬로 추가하세요.
- **메모리 계층 존중** — 새 SOP는 `memory/sops/` 에 markdown으로, L0/L1 헌법은 손대지 말기.
- 불필요한 의존성 추가 자제 (GenericAgent의 가치는 "최소 의존").

## 5. 커밋 메시지 컨벤션

본 레포의 git log를 보면 다음 prefix가 일관되게 쓰입니다.

| prefix | 용도 |
|---|---|
| `feat:` | 새 기능 |
| `fix:` | 버그 수정 |
| `refactor:` | 동작 변경 없는 구조 개선 |
| `docs:` | 문서만 수정 |
| `i18n:` | 다국어 |
| `cleanup:` | 데드코드 / 정리 |

스코프를 괄호로: `feat(tg): ...`, `fix(stapp): ...`, `fix(wechatapp): ...`.

```text
feat(tg): stream Telegram responses by turn with summaries (#208)
```

## 6. PR 체크리스트

- [ ] `python launch.pyw` 가 에러 없이 뜬다
- [ ] 영향 받는 프론트엔드(들)에서 직접 채팅으로 검증
- [ ] mykey.py / 토큰 / `.env` 류 시크릿이 diff에 들어가지 않았다
- [ ] PR 본문에 **재현 절차** 와 **변경 이유** 1–3줄
- [ ] 관련 이슈 번호 링크 (`Fixes #123`)

## 7. 메모리 SOP를 추가한다면

핵심 원칙 (메모리 시스템 SOP에서 차용):

1. **헌법(L0)은 건드리지 않는다** — 시스템 프롬프트의 메타 규칙.
2. **L1 Insight Index에 한 줄 등록** — 새 SOP는 이 인덱스에서 발견 가능해야 한다.
3. **action-verified axioms 만 적는다** — "내가 해보고 동작했다" 만 SOP가 된다. 추측은 금지.
4. **간결**: 한 SOP는 가능한 짧게, 페이지(`<200줄`) 안에서 끝낸다.

## 8. 기여하기 좋은 것들

- 새 LLM 백엔드 (DeepSeek/Qwen 등) — `llmcore.py` 에 세션 클래스 추가
- 새 프론트엔드 (Slack, Discord 등) — `frontends/` 에 `chatapp_common.AgentChatMixin` 활용
- 메모리 SOP / Skill — `memory/sops/`, `memory/persistent/`
- 문서 번역 / Mintlify 페이지
