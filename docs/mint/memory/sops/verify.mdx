## 언제 쓰는가
plan 모드 4단계 검증, 또는 임의의 산출물에 대해 적대적 검증이 필요할 때. 호출자가 명령을 무작위로 다시 실행해 검증할 수 있습니다.

## 작동 원리 (Why)
구현자(LLM)는 자기 코드를 검증할 때 mock과 happy path만 봅니다. 메인 agent는 확증 편향이 있어 표면적 성공에 속기 쉽습니다. 그래서 **독립 컨텍스트의 검증자**가 도구 호출 증거로 적대적 탐침을 던져야 진짜 검증입니다.

## 두 가지 실패 모드 (자가 점검)

1. **검증 회피**: 핑계로 실행 안 함 — 코드 읽기, "어떻게 될지" 서술, PASS 작성. **코드 읽기는 검증이 아닙니다.**
2. **앞 80%에 속음**: 통과한 테스트만 보고 PASS, 절반 기능이 빈 껍데기인 것 못 봄. **당신의 가치는 마지막 20%에 있습니다.**

## 철칙 (위반 → VERDICT 무효)

1. 반드시 **실행**할 것. 돌릴 수 있는 것은 돌리고, 볼 수 있는 것은 스크린샷.
2. 반드시 **도구 증거**가 있을 것. 도구 출력 없는 PASS = SKIP.
3. **독립 검증**. 테스트 스위트는 컨텍스트지 증거가 아닙니다.

## 산출물 유형별 검증 동작

| 산출물 | 필수 |
|---|---|
| 웹페이지/프런트엔드 | 열기 + 스크린샷 → console 에러 → curl 하위 자원 |
| 스크립트/CLI | 실행 → stdout/stderr/exit code → 경계 입력 재실행 |
| 데이터 파일 | 형식 검사 → 행 수 → 첫/중/끝 3건 추출 |
| API/서비스 | endpoint 호출 → 응답 형태 (200만 보지 말고) → 잘못된 입력 |
| 설정/문서 | file_read 전체 → 문법 → 기존 깨짐 여부 |
| Bug 수정 | 원래 bug 재현 → 수정 검증 → 회귀 테스트 |
| 일괄 조작 | 총수 → 첫/중/끝 추출 → 중복/누락 |

## 적대적 탐침 (최소 하나 실행)
경계값 (0/빈/초장/유니코드) · 멱등성 (같은 조작 두 번) · 의존성 누락 · 고아 ID

## 사용 예시

```text 출력 형식
| # | 검증 동작 | 도구 | 핵심 출력 요약 | PASS/FAIL |

각 점검: Command run → Output observed → Result

최종 판정 (리터럴, 변형 없음):
- VERDICT: PASS    — 핵심 점검 통과
- VERDICT: FAIL    — 미해결 문제 (재현 단계 첨부)
- VERDICT: PARTIAL — 환경 제한 검증 불가만 (이유 명시)
```

## 자주 빠지는 함정

<Warning>
**합리화 핑계 식별**:
- "코드가 맞아 보여" → 실행하세요.
- "테스트가 이미 통과했어" → 구현자가 LLM. 독립 검증.
- "별 문제 없을 거야" → "없을 거야" ≠ "검증됨". 실행하세요.
- "브라우저/도구가 없어" → 사용 가능한 도구를 점검했나요?
</Warning>

<Tip>
**자가 점검**: 도구 호출 대신 설명을 쓰고 있나요? 멈추세요. 도구를 호출하세요.
</Tip>
