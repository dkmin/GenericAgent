## 언제 쓰는가
대량 코드 읽기, 병렬 처리, 행동 검증, 또는 메인 컨텍스트를 깨끗이 유지하고 싶을 때.

## 작동 원리 (Why)
subagent는 **독립 컨텍스트**를 갖습니다. 문서 A 처리의 긴 컨텍스트가 문서 B 처리 품질을 오염시키는 것을 방지하고, 메인 agent의 가장 희소한 자원(컨텍스트)을 보존합니다.

## 파일 IO 프로토콜

- 디렉터리: `temp/{task_name}/` (cwd가 temp/면 `./{task_name}/`)
- 기동: `python agentmain.py --task {name} [--input "..."] [--bg] [--llm_no N]` (cwd = 코드 루트)
- `--input`은 자동으로 디렉터리 생성 + 옛 output 정리 + input.txt 작성. 긴 텍스트는 input.txt 수동 작성 후 `--input` 없이 기동.
- `--bg` 백그라운드, PID 출력 후 exit. 같은 code_run 안에서 sleep 후 poll 가능.
- 통신: output.txt (append, `[ROUND END]` = 라운드 완료) → reply.txt 작성으로 계속 → 10분 미작성 시 종료.
- 개입 파일: `_stop` (라운드 후 종료) | `_keyinfo` (working memory 주입) | `_intervene` (지시 추가)
- 감독 모드는 `--verbose` 추가로 도구 실행 결과를 output에 포함.

## 시나리오 1: 테스트 모드 (행동 검증)

agent의 실제 행동 관찰, RULES/L2/L3/SOP 보정.

**테스트 원칙**: 목표만 주고 위치 힌트/방법 유도 안 함, 자율 선택을 관찰.

**두 가지 테스트**:
- **SOP 품질 테스트**: input에 SOP명 지정 (예: `ezgmail_sop로 최근 3통 미열람 메일 확인`)
- **내비게이션 능력 테스트**: input에 목표만, subagent가 insight에서 올바른 SOP 자율 발견하는지

## 시나리오 2: Map 모드 (병렬 처리)

N개 독립 동형 하위 작업을 각자의 subagent에 분배.

**제약**:
- 파일시스템 공유 OK (다른 입력 → 다른 출력)
- 키보드/마우스는 공유 불가
- 브라우저는 동시 같은 탭 조작 회피

**표준 흐름 (map-reduce)**:
1. 메인이 데이터 덤프, 여러 독립 입력 파일로 저장
2. 파일별로 subagent 하나씩 기동
3. 모든 완료 대기, 메인이 각 출력 읽고 집계

## 시나리오 3: subagent 내부 plan_mode

3개 이상 하위 단계 + 의존성 → subagent 내부에 `./subagent_plan.md` 생성.

**파일 전달**: 메인이 `context.json` 생성, 모든 파일 **절대 경로** 포함:

```json
{
  "task": "작업 설명",
  "work_dir": "/absolute/path/to/plan_dir/",
  "input_files": {"paper_info": "/abs/path/paper_info.txt"},
  "output_files": {"pdf": "/abs/path/paper.pdf", "report": "/abs/path/paper_report.md"},
  "dependencies": ["paper_info.txt 가 존재해야 함"]
}
```

<Warning>
- subagent 기동 후 **첫 단계는 반드시 context.json 읽기**
- 모든 파일 조작은 context.json 내 **절대 경로** 사용
</Warning>

## 자주 빠지는 함정

<Warning>
- **input에 단계/과한 설명 금지**: subagent도 동등한 지능, 목표 + 제약만으로 충분. 대량 데이터는 경로로 전달.
- **메인 유휴 시 무뇌 sleep 폴링 금지**: output 읽고 진행 관찰, 필요 시 개입 파일로 보정.
- **subagent cwd는 temp/**, task 디렉터리가 아님.
</Warning>
