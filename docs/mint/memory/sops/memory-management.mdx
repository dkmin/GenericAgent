## 언제 쓰는가
새 정보를 메모리에 기록하기 전, 또는 기존 메모리를 수정/리팩터링할 때 반드시 먼저 읽어야 하는 헌법.

## 작동 원리 (Why)
메모리 수정은 **영구적 손상**이며, 잘못은 매 라운드 복리로 누적됩니다. 함정 회피 지침이 잘못 압축되면, 그 후 모든 세션이 그 잘못을 상속받습니다. 그래서 L0가 **공리(axiom)** 형태로 먼저 옵니다.

## 핵심 공리 (Core Axioms)

### 1. 행동 검증 원칙 (Action-Verified Only)
L1/L2/L3에 기록되는 모든 정보는 **성공한 도구 호출 결과**여야 합니다. 모델의 "고유 지식", "추론 추측", "미실행 계획", "검증되지 않은 가설"을 사실로 기록하는 것은 엄격히 금지.

> **구호: No Execution, No Memory.**

### 2. 검증된 데이터의 신성불가침
행동을 통해 검증된 유효 설정·함정 회피 지침·핵심 경로는 리팩터링 시 **절대 폐기 금지**. 문장 압축은 가능하나 정확성과 추적 가능성은 잃지 않아야 합니다.

### 3. 휘발성 상태 저장 금지
타임스탬프, 임시 Session ID, 실행 중인 PID, 연결된 디바이스 정보 등은 저장 엄금.

### 4. 최소 충분 포인터
상위 계층은 하위 계층을 식별하는 **최단 식별자**만 둡니다. 한 단어라도 더하면 중복.

## L1 ↔ L2/L3 동기화 규칙

| 조작 | L1 동기화 |
|------|--------|
| L2/L3 신규 상황 추가 | 신규는 기본 저빈도 → L3 리스트에 파일명 추가 |
| L2/L3 상황 삭제 | 해당 키워드/매핑 줄 삭제 |
| L2/L3 값 수정 | 상황 위치에 영향 없으면 L1 미수정 |
| 일반 함정 회피 패턴 발견 | 한 줄로 압축해 RULES에 추가 |

<Warning>
**동기화 레드라인**: L1에는 키워드/이름만, 세부 옮기기 금지. 괄호 안에는 상황 트리거 단어(2-4자)만, 메커니즘/방법/단계 기재 금지.

반례: ❌ `sop_name(상황A:방법1+방법2)` → ✅ `sop_name(상황A)`
</Warning>

## 사용 예시

<CodeGroup>
```text 새 도메인 SOP 추가 시
1. memory/foo_sop.md 작성 (L3)
2. L1 (global_mem_insight.txt)에 트리거 단어 한 줄 추가:
   foo_sop(직관에 반하는 트리거 단어)
3. L1 총 줄 수 ≤30 검증
```

```text L2 신규 사실 추가 시
1. global_mem.txt 에 ## [SECTION] 추가
2. L1의 해당 TOPIC 내비게이션 줄 갱신 (내비게이션만)
```
</CodeGroup>

## 자주 빠지는 함정

<Warning>
- **L1 overwrite 금지**: 단어 수준 patch만 허용. 통째 다시 쓰는 순간 검증된 함정 지침이 증발할 수 있음.
- **번역/내용 설명 추가 금지**: `proxy-pool/(代理池)` 같은 이름 번역, `opencli_sop(66站点CLI,复用Chrome session)` 같은 구현 세부는 모두 군더더기.
- **L1에 "How to" 작성 금지**: L1은 존재성 인코딩만, 방법은 L3에.
</Warning>
