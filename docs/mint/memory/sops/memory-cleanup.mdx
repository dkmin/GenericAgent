## 언제 쓰는가
L1(`global_mem_insight.txt`)이 30줄을 넘기 시작했거나, 잘못된 트리거 단어로 SOP 발견이 안 될 때.

## 작동 원리 (Why)
LLM 자체가 **압축기 + 디코더**입니다. L1은 단지 어떤 종류의 지식이 **존재한다**는 것만 알려주면 되고, 그러면 LLM이 tool call로 깊은 내용을 가져옵니다.

> **L1의 본질**: 가능한 가장 짧은 단어 수로 — 어떤 상황에서 어떤 메모리가 사용 가능한지(존재성)를 표현.

## ROI 평가
```
ROI = (이 단어들을 안 둘 때의 실수 확률 × 비용) / 매 라운드 단어 비용
```

L1은 두 종류 콘텐츠를 두며 통일된 ROI로 평가:
- **존재성 포인터**: L2/L3 지식을 가리키는 가장 짧은 트리거 단어
- **행동 규칙**: 알려주지 않으면 저지르는 실수 (치명적·고빈도)

## 빠른 판단

### 남길 것
직관에 반하는 트리거 단어 — 힌트가 없으면 SOP를 찾아볼 생각이 들지 않는 상황 단어.

> 예: `tmwebdriver_sop(httponly cookie)` — `httponly cookie` 단어가 없으면 cookie 가져올 때 tmwebdriver를 찾아봐야 한다는 생각이 안 듦.

### 지울 것
- **이름 번역**: `proxy-pool/(代理池)` → 이름 자체가 자명, 괄호 군더더기
- **내용 설명**: `opencli_sop(66站点CLI,复用Chrome session)` → 구현 세부는 SOP 내부 사항
- **직관적으로 떠오르는 능력**: 알려주지 않아도 떠오름 → 이득 0
- **중복**: L3가 이미 커버하는 규칙

## 압축 4원칙

1. **이름이 자명하면 > 설명 붙이는 것보다 낫다**: SOP 이름으로 충분히 표현되면 L1에 주석 달지 않음. 이름 변경의 ROI가 L1 수정의 ROI보다 높을 때가 많음.
2. **존재성 집합의 최소 기술**: 비슷한 항목이 같은 상위 상황으로 묶이면 집합명만. 예: `qq조작/feishu조작/wecom조작` → `im조작:*_im_sop`.
3. **항목 = 상황↔방안 존재성**: 괄호 안에는 **직관에 반하는 트리거 단어만**. 단순 번역/내용 설명/구현 세부는 모두 낭비.
4. **계층 분할 정렬**: 행동 규칙·고ROI 항목은 위쪽, 순수 존재성 포인터는 평탄 리스트로.

## 정리 절차

1. L1을 한 줄씩 읽고 `|`로 조각 분할, 분류: 존재성 포인터 / RULES / 번역 / 내용 설명 / 구현 세부 / 중복
2. **RULES부터 정리**: "전역 고ROI인가, 특정 상황 저위험인가?" 자문
   - 전역 고ROI → 유지
   - 특정 상황 / 저위험 → L3로 강등 또는 삭제
3. **존재성 포인터 정리**: 상황↔방안 존재성 점검. 직관에 반하는 트리거 단어만 추가.
4. **L3 파일명이 자명한지 확인**. 이름 변경으로 해결 가능하면 L1에 설명 추가하지 말고. 마지막으로 총 줄 수 ≤ 30 검증.

## 자주 빠지는 함정

<Warning>
**레드라인**: 메모리 수정은 영구적 손상이며, 잘못은 매 라운드 복리로 누적됩니다. **L1은 단어 수준 patch만 허용, overwrite 금지**. 오해 유발 시 즉시 L1 수정 또는 메모리 이름 변경.
</Warning>
