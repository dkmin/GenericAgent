# 메모리 정리 SOP

## 핵심 원칙: 존재성 인코딩
LLM 자체가 압축기 + 디코더입니다. L1 은 단지 **어떤 종류의 지식이 존재한다는 것을 인지**시키기만 하면 되고, 그러면 LLM 이 tool call 로 깊은 내용을 직접 가져옵니다.

**L1 의 본질: 가능한 가장 짧은 단어 수로 — 어떤 상황에서 어떤 메모리가 사용 가능한지(존재성)를 표현.**

L1 은 두 종류 콘텐츠를 두며, 통일된 ROI 로 평가합니다:
- **존재성 포인터**: L2/L3 지식을 가리키는 가장 짧은 트리거 단어
- **행동 규칙**: 알려주지 않으면 저지르는 실수 (치명적이거나 고빈도, ROI 가 임계치를 넘기만 하면 됨)

ROI = (이 단어들을 안 둘 때의 실수 확률 × 비용) / 매 라운드 단어 비용

## 빠른 판단
**남길 것**: 직관에 반하는 트리거 단어 — 힌트가 없으면 SOP 를 찾아볼 생각이 들지 않는 상황 단어. 예: `tmwebdriver_sop(httponly cookie)` — `httponly cookie` 라는 단어가 없으면 cookie 를 가져올 때 tmwebdriver 를 찾아봐야 한다는 생각이 들지 않습니다
**지울 것**:
- 이름 번역: `proxy-pool/(代理池)` → 이름 자체가 자명, 괄호는 군더더기, `proxy-pool` 만으로 충분
- 내용 설명: `opencli_sop(66站点CLI,复用Chrome session)` → 구현 세부는 SOP 내부 사항이지, 트리거 상황이 아님
- 직관적으로 떠오르는 능력: 알려주지 않아도 떠오름 → 이득 0, 매 라운드 비용만 헛소비
- 중복: L3 가 이미 커버하는 규칙 / L1 의 다른 줄에 이미 포함된 조각

## 압축 4원칙
1. **이름이 자명하면 > 설명을 붙이는 것보다 낫다**: SOP 이름으로 충분히 표현되면 L1 에 주석을 달지 않습니다. 이름 변경의 ROI 가 L1 수정의 ROI 보다 높을 때가 많습니다
2. **존재성 집합의 최소 기술**: 비슷한 항목 여러 개가 같은 상위 상황으로 묶이면, 집합명으로 그 능력군의 존재를 표현하고 하위 항목을 펼치지 않습니다. 예: `qq조작/feishu조작/wecom조작` → `im조작:*_im_sop`. 하위 이름이 자명하면 이름만 나열하고 번역하지 않습니다
3. **항목 = 상황↔방안 존재성**: 예: `영상이해:yt-dlp로 자막 추출`, `fofa(자산매핑)` — 상황명이 트리거 단어, 방안명이 존재성 인코딩. 괄호 안에는 **직관에 반하는 트리거 단어만** 둡니다. 직관적이지 않은 것 (단순 번역/내용 설명/구현 세부) 은 모두 낭비입니다
4. **계층 분할 정렬**: 행동 규칙이나 고빈도 고ROI 항목은 위쪽 상황 줄에, 순수 존재성 포인터는 L2/L3 평탄 리스트로

## 정리 절차
1. L1 을 한 줄씩 읽고 `|` 로 조각 분할, 분류부터: 존재성 포인터 / RULES / 번역 / 내용 설명 / 구현 세부 / 중복
2. RULES 부터 정리: 항목별로 "이게 전역 고ROI 인가, 특정 상황의 저위험 규칙인가?" 자문
   - 전역 고ROI → 유지
   - 특정 상황 / 저위험 → L3 로 강등 또는 삭제
3. 다음 존재성 포인터 정리: **상황↔방안 존재성**을 표현하는지 점검. 상황 트리거 단어는 **직관에 반하는** 경우만 추가, 번역/내용 설명/구현 세부는 삭제
4. L3 파일명이 자명한지 확인. 이름 변경으로 해결 가능하면 L1 에 설명을 추가하지 말고. 마지막으로 총 줄 수 ≤ 30 검증

**레드라인**: 메모리 수정은 영구적 손상이며, 잘못은 매 라운드 복리로 누적됩니다. L1 은 단어 수준 patch 만 허용, overwrite 금지
오해를 유발하면 즉시 L1 을 수정하거나 메모리 이름을 변경합니다
