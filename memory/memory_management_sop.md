## 0. 핵심 공리 (Core Axioms - 최우선)
1.  **행동 검증 원칙 (Action-Verified Only)**
    *   **정의**: L1/L2/L3 에 기록되는 모든 정보는 **성공한 도구 호출 결과**(예: `shell` 실행 성공, `file_read` 로 내용 확인, 코드 실행 통과 등)에서 비롯되어야 합니다.
    *   **금지**: 모델의 "고유 지식", "추론 추측", "미실행 계획", "검증되지 않은 가설"을 사실로 기록하는 것을 엄격히 금지합니다.
    *   **구호**: **No Execution, No Memory. (행동 없이 메모리 없음)**
2.  **검증된 데이터의 신성불가침 (Sanctity of Verified Data)**
    *   **정의**: 행동을 통해 검증된 유효 설정, 함정 회피 지침, 핵심 경로는 리팩터링(Refactoring/GC) 시 **절대 폐기 금지**.
    *   **조작**: 문장 압축 가능, 계층 이동 가능 (L2 에서 L3 로), 그러나 정보의 정확성과 추적 가능성은 잃지 않아야 합니다.
    *   메모리 수정 시 극도로 신중할 것. 가능한 한 overwrite 나 code run 을 피하십시오. 소량의 patch 만 허용, 수정이 어려우면 차라리 손대지 마십시오.
3.  **휘발성 상태 저장 금지 (No Volatile State)**
    *   **정의**: 시간/세션에 따라 자주 바뀌는 데이터의 저장을 엄격히 금지합니다.
    *   **예**: 현재 타임스탬프, 임시 Session ID, 실행 중인 PID, 특정 절대 경로, 연결된 디바이스 정보
4.  **최소 충분 포인터 (Minimum Sufficient Pointer)**
    *   상위 계층은 하위 계층을 식별할 수 있는 최단 식별자만 둡니다. 한 단어라도 더하면 중복.

## 메모리 계층 아키텍처
```
L1: global_mem_insight.txt (극간결 인덱스 계층 - 엄격히 ≤30줄)  
    ↓ 내비게이션 포인터 (Pointer)  
L2: global_mem.txt (사실 라이브러리 계층 - 현재는 짧지만 팽창함)  
    ↓ 상세 참조 (Reference)  
L3: ../memory/ (기록 라이브러리 계층 - .md/.py 등 다양한 파일 포함)  
L4: ../memory/L4_raw_sessions/ (이력 세션 계층 - scheduler reflect 가 자동 수집, 과거 컨텍스트 위치 추적 가능)  
```

## 각 계층 책임과 원칙
### L1: 글로벌 메모리 인덱스 (global_mem_insight.txt)
**책임**: L2 와 L3 에 극간결 내비게이션 인덱스 제공, 핵심 능력이 발견 가능하도록 보장.
**특징**:
- 크기 제한: ≤ 30 줄 (하드 제약), < 1k tokens (희망). 세부사항 기재 엄금 (극히 고빈도 작업 제외)
- 내용: 두 계층의 「상황 키워드 → 메모리 위치」 매핑 + RULES (레드라인 규칙 + 고빈도 실수 포인트)
  - 1계층: 고빈도 상황 key → value (sop/py/L2 section 명을 직접 제공), 자명한 이름은 한 단어만 쓰고 번역 반복 안 함
  - 2계층: 저빈도 상황은 키워드만 나열, 필요 시 L2 read 또는 ls L3 로 직접 위치
  - 핵심: 상황 트리거 단어는 매우 중요 (인덱스 없으면 능력 존재를 모름) 하지만 How-to 세부 기재 금지
  - RULES: 압축형 함정 회피 지침, 포함:
    - 레드라인 규칙 (치명형): 위반 시 프로세스 종료/시스템 크래시 (예: `무조건 python kill 금지(자기 자신을 죽임)`)
    - 레드라인 규칙 (은밀형): 위반해도 에러 없으나 잘못된 결과 생성 (예: `검색은 google, baidu 금지`)
    - 고빈도 실수 포인트: 잊기 쉬운 핵심 제약 (예: `es(PATH 에 있음)` 경로 검색 방지)
- 갱신: L2/L3 신규/삭제 시 빈도를 판정해 해당 계층에 배치. 수정 시 극도로 신중. overwrite 나 code run 비허용. 소량 patch 만, 수정이 어려우면 손대지 않음.
**금지**: 비밀번호, API Key 기록 엄금. 비민감 트리거 파라미터 (예: 프록시 포트) 인라인 허용. "How to" 또는 상세 설명 작성 금지. 특정 작업의 기술 세부 포함 엄금 (특정 작업 세부는 L3 에 두어야 함). 로그 기록 작성은 더더욱 엄금!
---
### L2: 글로벌 사실 라이브러리 (global_mem.txt)
**책임**: 글로벌 환경적 사실 (경로, 인증정보, 설정, 상수 등) 저장.
**특징**:
- 추세: 환경 확장에 따라 팽창 (수용 가능)
- 내용: `## [SECTION]` 으로 조직된 사실 항목들
- 동기화: 변경 시 L1 의 해당 TOPIC 내비게이션 줄 갱신, 내비게이션만 가능
**금지**: 휘발성 상태 저장 금지, 추측 저장 금지, 대모델이 추론 가능한 일반 상식 저장 엄금
---
### L3: 작업 단위 간결 기록 라이브러리 (../memory/)
책임: L1/L2 에 담을 수 없지만 **특정 작업**의 향후 재사용에 결정적인 소량의 상세 정보 보충. 내용은 재사용 요구 충족 전제하에 **가능한 한 짧게**.
원칙:
- 다음만 기록: 세션을 넘어 중요하고, 소량의 file_read / web_scan / 간단 스크립트로 빠르게 재구성하기 어려운 핵심
- 우선 기록: 해당 작업 특유의 숨겨진 사전 조건, 전형적 함정. 잊으면 고비용 재시도를 유발하는 정보
- 기록 안 함: 일반 절차, 몇 단계 탐색으로 다시 얻을 수 있는 경로 또는 상태 정보
형식:
- SOP (*_sop.md): 단일 작업 또는 소규모 작업군에 대한 극간결 「핵심 사전조건 + 전형적 함정」 체크리스트, 장문 튜토리얼 회피
- 도구 스크립트 (*.py): 재사용성 높고 로직이 비교적 복잡하며 매번 재추론을 원치 않는 처리 흐름만 캡슐화

## L1 ↔ L2/L3 동기화 규칙
| 조작 | L1 동기화 |
|---------|--------|
| L2/L3 신규 상황 추가 | 신규는 기본 저빈도 → L3 리스트에 파일명 추가 (자명하면 설명 없이, 직관에 반하는 상황만 괄호로 트리거 단어 추가) |
| L2/L3 상황 삭제 | 해당 계층의 키워드/매핑 줄 삭제 |
| L2/L3 값 수정 | 상황 위치에 영향 없으면 L1 미수정 |
| 일반 함정 회피 패턴 발견 | 한 줄로 압축해 RULES 에 추가 |

> **동기화 레드라인**: L1 에는 키워드/이름만, 세부 옮기기 금지. 괄호 안에는 상황 트리거 단어 (2-4 자) 만, 메커니즘/방법/단계 기재 금지. L1 의 token 수와 인덱스 효용을 평가해야 합니다.
> 반례: ❌ sop_name(상황A:방법1+방법2+방법3) → ✅ sop_name(상황A)

## 정보 분류 빠른 결정 트리
```
"이 정보는 어느 계층에?"

『환경 특이적 사실』인가? (IP, 비표준 경로, 인증정보, ID, API 키 등, 대모델 Zero-shot 으로 정확히 생성 불가)
  ├─ YES → L2 (global_mem.txt)
  │        그 다음 → 빈도에 따라 L1 1계층 (key→value) 또는 2계층 (키워드만) 에 배치
  │
  └─ NO
       ↓
       『일반 조작 패턴』인가? (전역적 함정 회피 지침, 진단 방법, 특정 작업과 무관한 일반 준칙)
       ├─ YES → L1 [RULES] (압축된 1줄 준칙만)
       │
       └─ NO
            ↓
            『특정 작업 기술』인가? (어렵게 성공한, 향후 다시 쓸 작업, 예: 위챗 파라미터 파싱, 특정 게임 좌표, 임시 도구 설정)
            ├─ YES → L3 (../memory/ 의 전용 SOP 또는 스크립트)
            │
            └─ NO → 『일반 상식』 또는 『중복 정보』로 판정: 저장 엄금, 즉시 폐기
```
