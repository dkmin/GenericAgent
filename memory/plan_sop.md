# Plan Mode SOP

**트리거**: 3단계 이상 의존성 / 다중 파일 협업 / 조건 분기 / 병렬 필요 | **금지**: 1-2단계 단순 작업은 바로 처리
작업 시작 전 반드시 작업 디렉터리 `./plan_XXX/` 를 먼저 생성합니다 (XXX = 작업의 영문 약명)
별도의 code_run({'inline_eval':True, 'script':'handler.enter_plan_mode("./plan_XXX/plan.md")'}) 한 번으로 plan 모드 진입

---

## 1. 탐색 단계 (계획 전 사전 단계, 필수)

⛔ **하드 룰 (먼저 읽고 행동)**:

- **메인 agent 가 환경 탐색을 직접 수행 금지** (반드시 subagent 에 위임, 예외 없음)
- 메인 agent 는: 디렉터리 생성, SOP 매칭, subagent 기동, 결론 읽기만 수행
- subagent 는 읽기 전용 탐색만, 어떤 파일도 수정하지 않으며 부작용 있는 조작 금지
- **탐색 subagent 기동 실패 시: 원인 진단 → 재시도, 최대 2회. 메인 agent 가 직접 탐색으로 폴백 금지**

**목표**: 어떤 계획도 쓰기 전에 3가지를 명확히:
① 환경 현황 (있는 것, 없는 것) ② 사용 가능한 SOP ③ 핵심 불확실성

**왜 반드시 subagent 인가**: 메인 agent 의 컨텍스트는 가장 희소한 자원. 탐색의 긴 출력이 계획 실행 공간을 차지합니다.

### Step 1: 디렉터리 생성 (필수) + SOP 매칭 + plan 플래그 설정 (메인 agent 가 직접)

1. 작업 디렉터리 생성 `mkdir plan_XXX/`
2. 컨텍스트 내 L1 Insight 인덱스에서 사용 가능한 도메인 SOP 매칭
3. checkpoint 갱신: `[작업] XXX | [요구사항] 한 줄 | [제약] 핵심 제한 | [매칭 SOP] ... | [진행] 탐색 단계`

### Step 2: 탐색 subagent 기동 (감독 모드)

subagent.md 에 따라 탐색 subagent 를 기동, **`--verbose` 추가**하여 감독 모드 활성화. input 요점:

- **작업**: 환경 정보 탐색하여 `plan_XXX/exploration_findings.md` 에 작성
- **탐색 항목** (작업 유형에 따라 선별, 전부 아님):
  - 코드류 → 핵심 파일 구조, 의존성, 진입점
  - 브라우저류 → 대상 페이지의 현재 상태, 상호작용 가능 요소
  - 자동화류 → 환경 점검 (which/pip/경로/권한)
  - 데이터류 → 표본 데이터 (앞 5행 + 뒤 5행 + 총량)
- **출력 형식**: `## 환경 현황` / `## 핵심 발견` / `## 위험/불확실성`
- **제약**: 읽기 전용 탐색, 파일 수정 금지, 도구 호출 ≤10회
- **복잡도 평가**: 탐색 시 데이터 규모 (파일 수, 행 수, 페이지 수) 를 기록해 findings 에 반영, 계획 시 위임 판단에 사용

### Step 3: 감독 대기 + 결론 읽기

메인 agent 는 output.txt 의 진행을 능동적으로 관찰합니다 (`--verbose` 출력은 원시 도구 결과를 포함). 무뇌 sleep 폴링이 아닙니다:

1. **관찰**: output.txt 를 읽고 subagent 의 탐색 방향과 원시 데이터 검토
2. **편차 보정** (필요 시):
   - 방향이 빗나감 → `_intervene` 작성으로 지시 추가
   - 핵심 컨텍스트 누락 → `_keyinfo` 작성으로 정보 주입
   - 충분한 정보 확보 → `_stop` 작성으로 조기 종료, 라운드 절약
3. **수거**: `[ROUND END]` 대기 후 `exploration_findings.md` 읽기

**산출**: `exploration_findings.md` (구조화된 발견 보고서). 메인 agent 는 이를 바탕으로 계획 단계로 진입, plan.md 헤드의 「탐색 발견」 섹션에 작성. 메인 agent 가 감독 과정에서 얻은 1차 인지도 계획에 직접 활용 가능.

## 2. 계획 단계 (검토 게이트 포함)

### Step 4: 도메인 SOP 읽기 → plan.md 작성

먼저 탐색 단계에서 매칭한 SOP 를 읽고 plan 골격을 작성. "⚠확인 필요" 허용, "조사 미완"을 핑계로 미루는 것 금지.

**[D] 위임 표시 규칙**: 각 단계 작성 시 탐색 발견과 결합해 작업량을 평가, 다음 중 어느 하나라도 부합하면 `[D]` 표시:

- 대량 코드/파일 읽기 필요 (예상 파일 >3개 또는 >100줄)
- 웹 페이지 탐색 후 정보 추출 필요
- 반복 작업 3회 이상 실행 필요
- 테스트/빌드 실행 후 출력 분석 필요

`[D]` 미표시 경우: plan.md 읽기/갱신, 단일 파일의 작은 수정, ask_user, 간단한 일회성 명령

**plan.md 형식**:

```markdown
<!-- EXECUTION PROTOCOL (매 라운드 필독, 당신의 실행 가이드)
1. file_read(plan.md), 첫 번째 [ ] 항목을 찾기
2. 해당 단계에 SOP 표시 → 그 SOP 의 🔑 빠른 참조 섹션을 file_read
3. 단계 실행 + Mini 검증 산출
4. file_patch 로 [ ] → [✓]+간단 결과 표시, 그 후 1단계로 돌아가 다음 [ ] 진행
5. 모든 단계 (검증 단계 포함) 완료 표시 후 → 종료 점검: file_read(plan.md) 로 [ ] 잔존 0개 확인
⚠ 기억으로 실행 금지 | 검증 단계 건너뛰기 금지 | 종료 점검 없이 종료 금지 | 멈춰서 순수 텍스트 보고 금지
💡 노가다(대량 코드/파일/웹/반복 작업)는 subagent 위임 우선, 메인 agent 컨텍스트를 깨끗이
-->
# 작업 제목
요구사항: 한 줄 | 제약: 핵심 제한

## 탐색 발견
- 발견1: XXX (출처: file_read/web_scan/code_run)
- 발견2: YYY
- 불확실성: ZZZ

## 실행 계획
1. [ ] Step 1 요약
   SOP: xxx_sop.md
2. [D] Step 2 요약 (subagent 위임 실행)
   SOP: yyy_sop.md
   의존: 1
3. [P] Step 3 요약 (병렬, subagent.md 읽고 Map 모드 실행)
   SOP: yyy_sop.md
4. [?] Step 4 (조건 분기)
   SOP: (없음) ← 고위험
   조건: X 성공 → 4.1, 아니면 → 4.2

---

## 검증 체크포인트
N+1. [ ] **[VERIFY] 독립 검증 subagent 기동**
     SOP: verify_sop.md plan_sop.md
     조작: plan_sop.md 4장 읽기 → verify_context.json 준비 → 검증 subagent 기동 → VERDICT 읽기 → 결과에 따른 처리
     ⚠ 건너뛰기 금지, subagent 미기동 상태에서 [✓] 표시 금지

---
```

### Step 5: 자체 점검 체크리스트 (메인 agent 가 항목별로 점검)

- □ 탐색 발견이 plan 에 모두 반영되었는가? (핵심 제약 누락 없음)
- □ 각 단계의 SOP 표시가 합리적인가? (그 SOP 가 실제로 그 단계를 해결하는가?)
- □ 단계 간 의존성이 올바른가? (숨은 의존성이 누락되지 않았는가)
- □ 고위험 단계 (SOP: 없음/비가역) 에 명확한 실행 사상이 있는가?
- □ 단계 입자가 적절한가? ("모든 파일 처리" 금지, 반드시 구체 항목으로 펼침)
- □ **복잡한/번거로운 단계가 [D] 표시되었는가?** (대량 코드/웹/반복 작업은 반드시 subagent 위임)
- □ **"검증 체크포인트" 섹션이 포함되고 [VERIFY] 단계가 있는가? (필수, 강제 단계)**

### Step 6: 사용자 확인

ask_user 로 plan 확인을 받은 후에야 실행 단계로 전환. **⛔ 사용자 미확인 시 실행 금지.**

### Step 7: 실행 단계 진입

checkpoint 갱신: `[실행] plan.md | 현재: Step1 | ⚡[P] 표시 있으면 반드시 subagent.md 읽고 Map 모드 실행`

## 3. 실행 단계 루프

> **핵심 원칙: 연속 실행, 멈춰서 보고하지 않기.** 한 단계 끝나면 즉시 file_read(plan.md) 로 다음 `[ ]` 찾기, 전부 완료까지.

### 매 라운드 흐름

1. **plan 읽기** — `file_read(plan.md)` 로 첫 `[ ]` 항목 위치 파악
2. **SOP 읽기** — 해당 단계에 SOP 표시 있으면 먼저 그 SOP 를 file_read
3. **표시 확인** — `[D]` → 반드시 subagent 위임, 메인 agent 는 결과 요약만 받음. `[P]` → subagent_sop.md 읽고 Map 모드 실행. `[?]` 조건 → 조건 평가 후 분기 선택, 미선택 분기는 [SKIP] 표시
4. **실행** — 특수 표시 없는 단계는 메인 agent 가 직접 실행
5. **Mini 검증** — 산출물 존재 및 합리성을 빠르게 확인 (file_read 로 비어있지 않음 확인, exit code 점검 등)
6. **완료 표시** — `file_patch` 로 `[ ]` → `[✓ 간단 결과]` (진행 상황을 plan.md 에 기록)
7. **계속** — 즉시 1단계로 복귀, file_read(plan.md) 로 다음 `[ ]` 실행

### 종료 점검 (마지막 단계 표시 후, 건너뛰기 금지)

file_read(plan.md) 로 전문 스캔, 모든 단계 ([VERIFY] 포함) 가 `[✓]`/`[✗]` 인지, `[ ]` 잔존 0개인지 확인.
출력: `🏁 종료 점검: [총 단계수] 단계 모두 완료, 0개 [ ] 잔존 → 작업 종료`
누락 발견 시 → 계속 실행, 완료 주장 금지.

### ⚠ 실행 단계 금지

- **기억으로 실행 금지**: 새 단계 진행 전 반드시 `file_read(plan.md)`, "다음 단계는 ... 였던 것 같다" 금지
- **검증 단계 건너뛰기 금지**: [VERIFY] 단계는 강제이며 "작업 다 끝났으니" 라는 이유로 건너뛸 수 없음
- **종료 점검 없이 종료 금지**: 마지막 단계 표시 후 반드시 file_read 전문 스캔으로 [ ] 잔존 0개 확인하고 🏁 종료 확인 줄을 출력
- **멈춰서 순수 텍스트 보고 금지**: 한 단계 끝나면 즉시 file_read(plan.md) 로 계속, 진행 요약 출력 금지

### 💡 동적 위임 원칙

`[D]` 미표시 단계라도 실행 중 다음 상황 발견 시 subagent 위임을 능동적으로 판단:

- 컨텍스트 이해를 위해 대량 코드/파일 읽기 필요 (>3 파일 또는 예상 >100줄)
- 반복 시행착오 디버깅 필요
- 웹 페이지 탐색으로 정보 추출 필요

방법: subagent 를 기동해 구체 작업 완료 후 간결한 요약을 반환하게 하고, 메인 agent 는 요약을 바탕으로 계속 결정. 메인 agent 컨텍스트를 깨끗이 유지하는 것이 최우선.

## 4. 검증 단계 (subagent 독립 검증)

> 모든 단계가 [✓] 후 진입. **강제로** 독립 subagent 를 기동해 적대적 검증, 컨텍스트 오염 회피.

### 트리거 조건

- 모든 실행 단계가 `[✓]` 표시
- **모든 plan 모드 작업은 subagent 검증을 거쳐야 함** (메인 agent 는 확증 편향 있어 표면적 성공에 속기 쉬움)

### Step 8: 검증 컨텍스트 준비

`./plan_XXX/` 아래 `verify_context.json` 생성, 포함:

- task_description: 원본 작업 설명 (사용자 원문)
- plan_file: plan.md 절대 경로
- task_type: code|data|browser|file|system
- deliverables: 인도물 목록 (type/path/expected)
- required_checks: 필수 점검 목록 (check/tool)

**전달 항목**: 작업 설명, plan 경로, 인도물 명세, 필수 점검. **전달 금지**: 실행 과정, 디버깅 기록.

### Step 9: 검증 subagent 기동

subagent.md 표준 흐름에 따라 검증 subagent 기동, input 요점:

- **역할**: 당신은 독립 검증자, 작업은 적대적 검증 (인도물이 사용 불가능함을 증명)
- **첫 단계 강제**: file_read verify_sop.md 로 검증 SOP 완독
- **verify_sop.md 3절** 에 따라 task_type 에 맞는 검증 전략 선택 실행
- **모든 점검에 도구 호출 증거** 필요 (실제 실행, 서술 아님)
- **작업 설명**: (원본 작업 설명 채우기)
- **인도물 목록**: (deliverables 목록 채우기)
- **출력**: result.md 에 verify_sop.md 6절 형식으로 출력, 마지막 줄 `VERDICT: PASS / FAIL / PARTIAL`
- **제약**: 3 라운드 내 완료, 매 라운드 최소 1회 실제 도구 호출

동시에 verify_context.json 의 경로를 전달, subagent 가 직접 상세 컨텍스트를 읽도록 함.

### Step 10: 검증 결과 수집

output.txt 폴링하여 `[ROUND END]` 대기, 그 후 result.md 읽기:

1. **VERDICT 줄 찾기**: result.md 의 마지막 몇 줄 읽고 `VERDICT: PASS/FAIL/PARTIAL` 추출
2. **유효성 점검**: 모든 PASS 항목에 도구 호출 출력이 없으면 (서술만), 검증 무효로 보고 FAIL 처리
3. **결과별 처리**:
   - **PASS** → 작업 완료 마무리 진입
   - **FAIL** → 수정 루프 진입
   - **PARTIAL** → 메인 agent 가 수용 가능 판단 시 완료, 아니면 수정
   - **VERDICT 줄 없음** → output.txt 에서 핵심 정보 추출, 메인 agent 가 직접 PASS/FAIL 판단

**작업 완료 마무리** (검증 PASS 후 실행):

1. plan.md 의 `[VERIFY]` 단계를 `[✓]` 표시
2. checkpoint 갱신: `[완료] XXX 작업 | [산출] ... | [경험] ...`
3. 사용자에게 작업 완료 확인

**중요**: 검증 PASS 후에만 [VERIFY] 를 [✓] 표시하고 작업 완료 주장 가능. 검증 FAIL 시 수정 루프 진입 필요.

**Fallback**: subagent 가 result.md 를 산출하지 못한 경우 (turn 소진), output.txt 에서 VERDICT 핵심 정보 추출.

### 수정 루프 (FAIL 후)

FAIL → 구체 실패 항목 추출 → 실행 단계로 복귀 수정 (재계획 아님) → 수정 완료 → 검증 subagent 재기동 → FAIL-재시도 최대 2 라운드, 초과 시 ask_user 로 개입

수정 시:

1. FAIL 항목을 새 단계로 plan.md 에 추가 (`[FIX]` 표시)
2. 실패 항목만 수정, 이미 PASS 한 부분은 재작업 금지
3. 수정 완료 후 verify_context.json 재준비 (실패 항목만 포함)

### 특수 시나리오 처리

브라우저/키보드마우스/스케줄 작업 등의 시나리오: 메인 agent 가 조작을 수행하고 증거 (스크린샷/녹화/로그) 를 내보내기 → subagent 가 증거 파일 검증. **메인 agent 가 직접 PASS/FAIL 판단 금지**.

---

## 5. 실패 처리

1. **기록**: checkpoint 에 `step_X: [FAILED] 원인 (retry: N/3)`
2. **재시도**: 네트워크 타임아웃 → 자동 3회 재시도 (2s/4s/8s) | 설정 오류 → 사용자 문의 | 그 외 → [✗] 표시 후 건너뜀
3. **subagent 실패**: stderr.log 확인 → 명확한 에러는 메인 agent 가 수정 후 재기동 | 알 수 없는 에러는 1회 재시도 | 최대 2회 재기동
4. **의존성 전파**: 단계 실패 후 후속 의존 항목은 [SKIP] 표시
5. **plan 오류**: 계획 단계로 롤백, plan.md 수정 후 검토 게이트 재통과

## 강제 제약

- 각 항목은 독립적인 완료 판정 기준이 있어야 함
- "모든 파일 처리" 금지, 반드시 구체 항목으로 펼침
- 한 번에 하나만; 계획에 오류 있으면 계획 단계로 복귀해 수정
- 비가역 조작 전에 한 단계 더 검증
